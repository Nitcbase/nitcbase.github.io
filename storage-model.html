Storage Model

Disk Model

Nitcbase assumes that disk is a sequence of blocks and a block is a sequence of bytes. The disk consists of 8192 blocks and a block is of 2048 bytes. Hence there is a total of 16MB of storage. Disk blocks are indexed from 0 to 8191.

Block# 0-3 are reserved for storing Block allocation map where as Block# 4 is for storing Database Catalog. Block allocation map tells us whether a particular block is free or occupied and requires one byte per each block. Hence a total of 8192/2048=4 blocks are required for Block allocation map. Based on type of information stored in a block, a block can be of 5 types- 1. Record block, 2. Index block, 3.Database Catalog block, 4.Relation Catalog block, 5.Attribute catalog block.

Record block structure

A database in Nitcbase is a collection of relations, and each relation is a collection of records. The DBMS must store relations in the disk and the strategy is to store each relation in a set of blocks in the disk, organized as a linked list. Each block in such a linked list will be called a record block and is designed to store a fixed number of records. The block will also contains some meta data like indices of the left and right blocks in the linked list. Nitcbase fixes the size of a record as 256 bytes.

A record block stores the actual data records. The record block is divided into slots of size 256 Bytes each. Each slot stores a single record. Each block can have a maximum of 7 slots/records. In addition to data records, each block also stores meta data in preamble/header which occupies the first 256 Bytes of the record block.

First four bytes(0-3) of header are used to identify the type of block(REC/IND/DBCAT/RELCAT/ATTRCAT) where REC represents a record block. Next four bytes(4-7) are used for storing parent block pointer which has no significance for a record block and can be set to -1. Byte# 8-11 and 12-15 are used for storing Left and right block numbers respectively. Next four bytes are used for storing number of records currently stored in the block whose value can range from 0 to 7. Bytes 20-31 are reserved for future use.

Slotmap is used for knowing whether a slot is occupied or free. It is of 7 bytes, one for each slot. If a slot is free, its corresponding entry in slotmap will be 0 else it will be 1. Slotmap occupies 32-38 bytes of the header. Rest of the bytes in the header i.e., 39-255 are unallocated.

A Record can have a maximum of 16 attributes and an attribute can have a maximum of 16 Bytes. Each attribute is alloted a fixed size of 16 bytes and hence each record is alloted a fixed size of 256 bytes in disk.

Index block structure

Nitcbase uses the B-Tree data structure for indexing records in a relation. The DBMS maintains at least one index per relation (and possibly more on requirement). Corresponding to each index of a relation, the DBMS mantains a B-Tree to store the indices. Each node of a B-Tree is stored into a block in the disk. A disk block that stores a node of a B-Tree is called an index block. Each index block in Nitcbase is designed to store a maximum of 55 indices. An index is a 3-tuple (attribute value, block number, slot number). Each index in a B-Tree node also has an associated pair of left and right child block pointers. Hence, in addition to the indices, each index block is required to store a total of 56 child pointers. (Only 56 Child pointers are required because the right child of one index is the same as the left child of the next index).

An index block is divided into three parts. The first 32 bytes contain a header, the next 1760 bytes store the indices, the following 224 bytes store the child pointers and the last 32 bytes are unused. The header contains meta data similar to that of a record block. The first four bytes of the header stores the value IND. PBlock is the Block# of parent block in the B-Tree. LBlock and RBlock have no significance for an index block. #Entries field stores the actual number of indices stored in the block. The remaining space in the header are left unused.

Each entry in the index block is a 3-tuple (attribute value, block number, slot number) corresponding to the particular record. Block# and slot# occupy 4 bytes each and any attribute occupies 16 bytes. Thus each index requires 24 bytes. Eight bytes of unused space is maintained for future use. Thus a total of 32 bytes are allocated for a single index entry. Since an index block can store upto 55 indices, the total space required for storing indices is 55*32=1760. Each child pointer requires 4 bytes of storage and the total space for storing 56 child pointers works out to 56*4 = 224. The remaining space in the block (32 bytes) is left unused.

Catalog structures

A relational database has a number of catalogs for storing the meta data information about databases, relations, attributes, indexes etc. In addition to storing these catalogs in disk, a DBMS also caches some of them in memory for quick access. Such cache is called Catalog cache. Nitcbase uses three types of catalogs for storing the meta data.

  1. Database Catalog
  2. Relation Catalog
  3. Attribute Catalog

In an RDBMS, a relation refers to collection of records where each record is an n-tuple of attribute values. An attribute is a column or field in a relation. A database is a collection of relations.

Database Catalog

Nitcbase stores database catalog in Block# 4 of the disk. Every database has an entry corresponding to itself in database cataog. Since there are 62 entries in Database Catalog block, a maximum of 62 databases are permitted in total. Each entry in database catalog is of size 32 bytes. Database catalog has three attributes- DBName, RelCat_Index and #Relations. In each entry of database catalog, DBName(0-15 bytes) stores the name of the database, RelCat_Index(16-19 bytes) stores the Block# of relation catalog for the database and #Realtions(20-23 bytes) specify the current number of relations present in the database, next 8 bytes(24-31) are left unused. In addition to Database Catalog information, this block also stores meta data in preamble/header which occupies the first 96 Bytes of the Database Catalog block. The structure of which can be inferred from the figure.

The first four bytes of the header stores the value DBCAT which represents, the block is currently storing Database Catalog. PBlock, LBlock and RBlock have no significance for a Database Catalog block. #Entries field stores the actual number of Database Catalog entries stored in the block(which represents number of databases currently present in the system).Next 8 bytes(byte# 24-31) are reserved for future use, Slotmap occupies next 62 bytes(byte# 32-93)(can be seen in figure), where each byte corresponds to one entry in the database catalog. The remaining 2bytes of header(byte#- 94-95) are left unused.

Relation Catalog

Relation Catalog stores meta information regarding relations in a database. Each database has a relation catalog. Each entry in Relation Catalog is of size 64 bytes and has the fallowing attributes: RelName, #Attributes, AttrCat_Index, #Records and RecBlk_Index.In each entry, RelName(byte# 0-15) stores name of the relation, #Attributes(byte# 16-19) stores number of attributes of the relation, AttrCat_Index(byte# 20-23) stores the block number of Attribute Catalog of the relation, #Records(byte# 24-27) stores the number of records currently stored in the relation and FirstBlkInd (byte# 24-27) stores the block number of the first record blocks of the relation. Since record blocks are arranged as linked list, FirstBlkInd essentially stores head of the linked list, next 32 bytes of each entry(byte# 32-63) are kept free. Nitcbase allocates one block for a relation catalog, which contains 30 entries, Hence a database can have a maximum of 30 relations.

Relation Catalog block also stores the meta data about the block in preamble/header which occupies the first 128 Bytes of the Relation Catalog block. The structure of which can be inferred from the figure.

The first four bytes of the header stores the value RELCAT which represents the block is currently storing Relation Catalog. PBlock, LBlock and RBlock have no significance for a Relation Catalog block. #Entries field(byte# 16-19) stores the actual number of Relation Catalog entries stored in the block.byte# 20-31 are kept reserved for future use. Slotmap occupies next 30 bytes(byte# 32-61), where each byte corresponds to one entry in the Relation catalog. The remaining 1byte of header(byte#- 62) is left unused.

Attribute Catalog

Attribute Catalog stores meta information regarding every attribute of a relation. There is an attribute catalog corresponding to every relation. Each entry of attribute catalog has fallowing attributes: AttrName, AttrType and RootBlk_Index where Attrname is name of the attribute and Attrtype is data type of the attribute, Nitcbase permits three data types for attributes. They are integer, floating point numbers and strings(of maximum size 16). Nitcbase uses B-Tree for indexing. RootBlk_Index stores the root block number of the B-Tree if there is an index created on the attribute else it contains -1.size of Attribute Catalog is one block which contains 62 entries, so we can use maximum of 62 attributes for each relation, but effectively we use only 16 entries, as Nitcbase permits a maximum of 16 attributes per relation. details of same can be inferred from the figure.

just like all other block types, Attribute Catalog also stores the meta data about the block in preamble/header which occupies the first 96 Bytes of the Attribute Catalog block.

The first four bytes of the header stores the value ATTRCAT which represents the block is currently storing Attribute Catalog. PBlock, LBlock and RBlock have no significance for a Attribute Catalog block. #Entries field(byte# 16-19) stores the actual number of Attribute Catalog entries stored in the block.byte# 20-31 are kept reserved for future use. Slotmap occupies next 62 bytes(byte# 32-93), where each byte corresponds to one entry in the Attributes catalog. The remaining 2bytes of header(byte#- 94-95) is left unused.
